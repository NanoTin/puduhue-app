## 3. Convenciones de Base de Datos

### 3.1. Nomenclatura de Tablas
- Todo en **minúsculas sin separador**: `prodlechedetalle`, `fundosestanquesclientes`.
- Tablas LOG: `<tabla>log` → `empresaslog`, `prodlechelog`.
- Tablas HIST: `<tabla>hist` → `usuariosempresashist`.

### 3.2. Nomenclatura de Columnas
- PK: `<tabla>id` (INT AUTO_INCREMENT) → `empresaid`, `usuarioid`.
- FK: `<tablarelacionada>id` → `empresaid` en `fundos`.
- Vigencia: `<tabla>vig` o `<tabla>activo` (TINYINT(1)).
- Código ERP: prefijo `<modulo>_erp*` → `pl_erpestablecimientocod`.

### 3.3. Columnas de Auditoría (cuando `Audit Columns = TRUE`)
```
auditcreacionusuarioid      INT
auditcreaciondispositivo    VARCHAR(100)
auditcreacionip             VARCHAR(50)
auditcreacionfechahora      DATETIME (NOW())
auditedicionusuarioid       INT (NULL)
auditediciondispositivo     VARCHAR(100) (NULL)
auditedicionip              VARCHAR(50) (NULL)
auditedicionfechahora       DATETIME (NULL)
```

### 3.4. Columnas de Tablas LOG
```
logid                       INT AUTO_INCREMENT
<entidadid>                 INT (PK de la entidad)
logusuarioid                INT
logdispositivo              VARCHAR
logip                       VARCHAR
logfechahora                DATETIME
logtipo                     VARCHAR ('INSERT','EDIT','ANULAR','ELIMINAR')
logparamjson                TEXT (copia del p_in_json)
logregbkpjson               TEXT (estado previo; NULL en INSERT)
```

## 4. Stored Procedures

### 4.1. Firma Estándar
Todos los SP comparten la misma firma:
```sql
IN  p_in_json       LONGTEXT
IN  p_in_usuarioid  INT
IN  p_in_dispositivo VARCHAR(100)
IN  p_in_ip         VARCHAR(50)
OUT p_out_json      LONGTEXT
```

### 4.2. Nomenclatura

| Tipo                | Nombre                              |
|---------------------|-------------------------------------|
| Insertar            | `sp_<modulo>_insertar`              |
| Editar              | `sp_<modulo>_editar`                |
| Anular (baja lógica)| `sp_<modulo>_anular`                |
| Eliminar (DELETE)   | `sp_<modulo>_eliminar`              |
| Listar resumen      | `sp_<modulo>_listar_resumen`        |
| Listar detalle      | `sp_<modulo>_listar_detalle`        |
| Consulta por ID     | `sp_<modulo>_consulta_por_id_resumen` |
| Consulta por ID det.| `sp_<modulo>_consulta_por_id_detalle` |

### 4.3. Reglas Clave
- **Transacciones**: `BEGIN/COMMIT/ROLLBACK` se manejan **solo desde PHP** (`Database.php`), **nunca** dentro de los SP.
- **Validaciones**: los SP validan JSON no vacío, usuario existente/activo/no bloqueado, duplicados al insertar, existencia al editar/anular.
- **LOG**: SP de mantenimiento insertan en tabla LOG correspondiente.
- **p_out_json**: devuelve `status`, `message`, y opcionalmente `id` generado.

## 5. Fuente de Verdad

Los archivos CSV en `database/spec/` son la fuente de verdad para el modelo de datos:

| Archivo                  | Contenido                                      |
|--------------------------|-------------------------------------------------|
| `Tables.csv`             | Listado de tablas con flags (Audit, LOG, HIST)  |
| `Columns.csv`            | Columnas por tabla con metadatos de SP/listado  |
| `Audit Columns.csv`      | Definición de columnas de auditoría             |
| `Generic Log Table.csv`  | Plantilla de tablas LOG                         |

### Generadores
- `database/generate_tables_from_csv.php` → genera `tables/*.sql`.
- `database/generate_sp_from_csv.php` → genera `sp/*.sql`.

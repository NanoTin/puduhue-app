## 2. Problemas Detectados

### 2.1. Responsive
- El menú lateral no se comporta correctamente en resoluciones < 768px.
- Algunas pantallas de transacciones (especialmente `prodleche_crear.php`, `suplanimal_crear.php`) tienen layouts que no se adaptan a dispositivos pequeños.
- Las tablas de listado no tienen scroll horizontal en mobile.

### 2.2. Grids de Datos
- **Sin paginación**: todos los registros se cargan de una vez.
- **Sin ordenamiento** por columnas (click en header).
- **Lag de renderización** cuando la consulta devuelve muchos registros (DOM completo en una sola carga).
- Sin búsqueda/filtro inline en tabla.

### 2.3. Inconsistencia Visual
- Botones con diferentes variantes de Bootstrap (`btn-primary`, `btn-outline-success`, `btn-sm`, etc.) sin un criterio unificado.
- Algunos módulos usan `.container` y otros `.container-fluid`.
- Formularios no siguen un ancho estándar (algunos usan `col-md-6`, otros no).
- Feedback con `alert()` bloqueante mezclado con alertas Bootstrap.

## 3. Estándares Propuestos

### 3.1. CSS BEM (Block-Element-Modifier)
Adoptar BEM como estándar para todos los estilos personalizados del proyecto.

Referencia operativa para vistas PHP: `docs/frontend_php_patrones_reutilizables.md`.

```css
/* Bloque */
.pdh-card { }

/* Elemento */
.pdh-card__header { }
.pdh-card__body { }

/* Modificador */
.pdh-card--highlighted { }
.pdh-card__header--compact { }
```

**Prefijo del proyecto**: `pdh-` para evitar colisiones con Bootstrap.

### 3.2. Toasts como Único Mecanismo de Feedback
- **Éxito** (CRUD exitoso): toast verde, auto-dismiss en 4s.
- **Error** (validación/servidor): toast rojo, requiere cierre manual.
- **Advertencia**: toast amarillo, auto-dismiss en 6s.
- **Información**: toast azul, auto-dismiss en 4s.
- **Prohibido**: `alert()`, `confirm()` nativo de JS para feedback de operaciones.
- `confirm()` solo se permite para acciones destructivas (anular/eliminar).

### 3.3. Evaluación CSS: Bootstrap 5 vs Tailwind CSS

#### Opción A: Mantener Bootstrap 5
- **Pros**: Ya implementado, no requiere build pipeline, compatible con hosting.
- **Contras**: Personalización limitada, clases genéricas, difícil de mantener BEM puro.

#### Opción B: Tailwind CSS (compilado local)
- **Pros**: Máximo control, CSS mínimo en producción (purge), modernidad.
- **Contras**: Requiere build pipeline local (`npx tailwindcss`), curva de aprendizaje.
- **Viable en hosting compartido**: Sí — se compila en PC local y se sube solo el archivo CSS resultante.

> [!IMPORTANT]
> **Decisión pendiente**: Definir si se adopta Tailwind o se continúa con Bootstrap 5 + BEM. Esta decisión debe tomarse como un ADR antes de iniciar las mejoras visuales.

### 3.4. Grid de Datos
Opciones evaluadas:

| Librería        | Peso  | Paginación | Sort | Filtro | Server-side | Dependencias |
|-----------------|-------|------------|------|--------|-------------|--------------|
| DataTables      | ~85KB | ✅         | ✅   | ✅     | ✅          | jQuery       |
| AG Grid Free    | ~200KB| ✅         | ✅   | ✅     | ✅          | Ninguna      |
| Simple-DataTables| ~30KB| ✅         | ✅   | ✅     | ❌          | Ninguna      |
| Custom PHP      | 0KB   | ✅         | ❌   | ❌     | ✅          | Ninguna      |

> [!IMPORTANT]
> **Recomendación**: Implementar paginación **server-side** en los SP (con `LIMIT`/`OFFSET`) para los módulos con mayor volumen de datos, combinado con una librería ligera de grid en el frontend.

### 3.5. Responsive
- Definir breakpoints: `< 576px` (mobile), `576-768px` (tablet portrait), `768-992px` (tablet landscape), `> 992px` (desktop).
- Menú lateral: colapsable por defecto en mobile, con botón hamburguesa.
- Tablas: scroll horizontal en mobile con `table-responsive`.
- Formularios: stack vertical en mobile (`col-12`), side-by-side en desktop (`col-md-6`).

---

## 4. Reglas de Estilo (resumen para desarrollo)

1. Usar prefijo `pdh-` para clases CSS personalizadas.
2. Aplicar BEM para toda clase personalizada nueva.
3. No crear CSS inline (`style="..."`) excepto para estilos dinámicos generados por JS.
4. Un solo archivo CSS personalizado (o uno por módulo si el proyecto crece).
5. Feedback siempre mediante toasts; prohibir `alert()` para operaciones CRUD.
6. Todas las tablas con `.table-responsive` para mobile.
7. Formularios deben usar grid de Bootstrap con clases responsivas (`col-md-6`, `col-lg-4`, etc.).
